  តើការ Upload រូបសុទ្ធតែឆ្លងកាត់ Server Render មែនទេ តើមានវិធីសាស្រ្តណាមិនឆ្លងកាត់ទេដើម្បីកាត់បន្ថយHTTP Responses

  បច្ចុប្បន្ន រូបភាពដែលលោកអ្នកថត គឺត្រូវធ្វើដំណើរពីរដំណាក់កាល៖
   1. ដំណាក់កាលទី ១: ពីទូរស័ព្ទ (Client) ទៅកាន់ Server Render (Go Backend)។
   2. ដំណាក់កាលទី ២: ពី Server Render ទៅកាន់ Google Drive (តាមរយៈ Apps Script)។

  ផលវិបាក: វាធ្វើឱ្យទិន្នន័យ (Data) ដើរឆ្លងកាត់ Server Render ពីរដង ដែលនាំឱ្យចំណាយ Bandwidth ច្រើន និងអាចដើរយឺត ប្រសិនបើរូបភាពមានទំហំធំ។

  ---

  វិធីសាស្ត្រមិនឆ្លងកាត់ Server Render (Direct Upload Strategy)

  ដើម្បីកាត់បន្ថយ HTTP Responses និង Data នៅលើ Server យើងអាចប្រើវិធី "Direct to Cloud" ឬ "Direct to Apps Script"៖

  ១. ការបង្ហោះត្រង់ទៅកាន់ Google Apps Script (Direct Client-to-AppsScript)
  យើងអាចកែសម្រួលឱ្យកម្មវិធី (Frontend) ផ្ញើរូបភាព Base64 ទៅកាន់ Google Apps Script ដោយផ្ទាល់តែម្តង ដោយមិនចាំបាច់ឆ្លងកាត់ Go Backend ឡើយ។
   * លំហូរថ្មី: Frontend 📤 -> Apps Script (Google Drive) -> 📥 បាន Link រូបភាព -> Frontend 📤 -> Go Backend (ផ្ញើតែ Link ខ្លីមួយសម្រាប់ Update DB)។
   * អត្ថប្រយោជន៍: Server Render នឹងមិនចាំបាច់ទទួលរូបភាពធំៗទៀតទេ (ទទួលតែអក្សរ Link ខ្លីៗ)។ វាជួយសន្សំសំចៃ Data និងល្បឿនបានច្រើនបំផុត។

  ២. ការប្រើប្រាស់ Cloud Storage API (ដូចជា Cloudinary ឬ Firebase)
  ប្រសិនបើលោកអ្នកមានតម្រូវការប្រើរូបភាពក្នុងបរិមាណរាប់ម៉ឺនសន្លឹកក្នុងមួយថ្ងៃ យើងអាចប្រើ API របស់សេវាកម្មទាំងនេះ។ ពួកវាអនុញ្ញាតឱ្យយើងបង្ហោះរូបភាពពី Browser ទៅកាន់ Cloud ដោយផ្ទាល់ និងមានសុវត្ថិភាព។

  ---

  តើលោកអ្នកគួរជ្រើសរើសផ្លូវណា?

  ខ្ញុំសូមណែនាំ វិធីទី ១ (Direct to Apps Script) ព្រោះលោកអ្នកមានហេដ្ឋារចនាសម្ព័ន្ធ Google រួចជាស្រេចហើយ។ ដើម្បីធ្វើវាបាន យើងត្រូវ៖
   1. កំណត់ CORS ក្នុង Apps Script ដើម្បីឱ្យវាទទួលយកការហៅ (Call) ពី Browser របស់អ្នកដោយផ្ទាល់។
   2. កែសម្រួលកូដក្នុង React ឱ្យហៅទៅកាន់ Apps Script URL ជាមុនសិន រួចចាំយក Link មកផ្ញើឱ្យ Go Backend តាមក្រោយ។

  ចំណុចគួរប្រុងប្រយ័ត្ន: ប្រសិនបើបង្ហោះត្រង់ពី Browser នោះ Apps Script Secret Key របស់អ្នកអាចនឹងត្រូវបានគេឃើញក្នុង "Network Tab" នៃ Browser។ ប្រសិនបើប្រព័ន្ធនេះប្រើតែក្នុងរង្វង់បុគ្គលិកផ្ទៃក្នុង វាមិនមែនជាបញ្ហាធំដុំឡើយ
  ប៉ុន្តែវាជាចំណុចសុវត្ថិភាពមួយដែលត្រូវដឹង។

  តើលោកអ្នកចង់ឱ្យខ្ញុំជួយកែសម្រួលកូដឱ្យដើរតាមផ្លូវ Direct Upload (មិនឆ្លង Server Render) នេះដែរឬទេ?

ហេតុអ្វីនិយាយថា ផ្ញើរូបភាព Base64 ទៅ?



សុំទោសដែលខ្ញុំប្រើពាក្យបច្ចេកទេសនាំឱ្យមានការភាន់ច្រឡំ។ មូលហេតុដែលខ្ញុំនិយាយថាផ្ញើជា Base64 ព្រោះវាជាវិធីដែលស្រួល និងមានប្រសិទ្ធភាពបំផុតសម្រាប់ធ្វើការជាមួយ Google Apps Script តាមរយៈការហៅ (Call) ពីកម្មវិធី Browser (React)៖

  ១. មូលហេតុបច្ចេកទេស (Compatibility)
  Google Apps Script ទទួលទិន្នន័យតាមរយៈមុខងារ doPost(e)។ នៅពេលយើងផ្ញើចេញពី Browser ទៅកាន់វាផ្ទាល់៖
   * ការផ្ញើជា Binary (Raw File) គឺពិបាករៀបចំជាមួយ CORS និងការបម្លែងទិន្នន័យក្នុង Apps Script ឱ្យទៅជារូបភាពវិញ។
   * ផ្ទុយទៅវិញ ការបម្លែងរូបភាព WebP ទៅជា Base64 (អត្ថបទ/String) ធ្វើឱ្យយើងអាចដាក់វាបញ្ចូលក្នុង JSON រួមជាមួយព័ត៌មានផ្សេងៗ (ដូចជា Order ID, ឈ្មោះអ្នកដឹក) បានយ៉ាងងាយស្រួល។

  ២. ដំណើរការក្នុង Apps Script
  នៅផ្នែក Google Apps Script វាងាយស្រួលណាស់ក្នុងការអានកូដ Base64 នេះ៖

   1 // វាយកកូដអត្ថបទ Base64 មកបម្លែងជារូបភាពវិញក្នុងមួយបន្ទាត់
   2 var blob = Utilities.newBlob(Utilities.base64Decode(base64Data), "image/webp", fileName);
   3 folder.createFile(blob); // រួច ចុះនៅពេលរក្សាទុកក្នុង Google Drive ជាWebp ត្រូវទេ?

  ៣. ការចំណេញទំហំទិន្នន័យ (WebP + Base64)
  ជាធម្មតា Base64 ធ្វើឱ្យទំហំទិន្នន័យធំជាងមុនប្រហែល ៣៣%។ ប៉ុន្តែដោយសារយើងបានប្រើ WebP ដើម្បីបង្រួមរូបភាពឱ្យតូចខ្លាំងរួចទៅហើយ (ពី 5MB មកសល់ត្រឹម 150KB) នោះការកើនឡើង ៣៣% នៃ ១៥០KB គឺនៅតែ តូចខ្លាំង (ប្រហែល ២០០KB ប៉ុណ្ណោះ)។
   * ការផ្ញើអត្ថបទ ២០០KB គឺលឿនខ្លាំងណាស់សម្រាប់អ៊ីនធឺណិតបច្ចុប្បន្ន។

  សេចក្តីសន្និដ្ឋាន៖
  យើងប្រើ Base64 ព្រោះវាជា "ភាសាអក្សរ" ដែល Browser និង Google Apps Script យល់គ្នាបានច្បាស់បំផុត និងមិនងាយមានកំហុស (Error) ពេលបញ្ជូនទិន្នន័យ។

  ប្រសិនបើលោកអ្នកចង់ប្រើវិធី Direct Upload (មិនឆ្លង Server Render) យើងនឹងត្រូវប្រើ Base64 នេះផ្ញើត្រង់ទៅកាន់ URL របស់ Apps Script តែម្តង។ តើលោកអ្នកចង់ឱ្យខ្ញុំជួយរៀបចំកូដតាមវិធីនេះដែរឬទេ?_






បាទ ពិតជាត្រឹមត្រូវណាស់! នៅក្នុងកូដ App Script/code_gs_api.gs ដែលកំពុងប្រើប្រាស់នេះ គឺបានកំណត់ឱ្យបង្កើតឯកសារជាទម្រង់ WebP រួចជាស្រេច៖

  ត្រង់បន្ទាត់ទី 749 ក្នុង uploadImageToDrive:

   1 const blob = Utilities.newBlob(decodedData, mimeType, fileName);
   2 const file = folder.createFile(blob);

  ដំណើរការគឺបែបនេះ៖
   1. Frontend: បង្រួមរូបភាពជា WebP រួចបម្លែងជា Base64 ផ្ញើមក។
   2. Backend (Go): ទទួលយក Base64 និងបញ្ជាក់ប្រភេទឯកសារ (mimeType) ថាជា "image/webp" រួចបញ្ជូនបន្តទៅ Apps Script។
   3. Apps Script: យកទិន្នន័យនោះមកបង្កើតជា File ក្នុង Google Drive ដោយកំណត់ប្រភេទជា WebP តាមរយៈ variable mimeType ដែលទទួលបាន។

  លទ្ធផលក្នុង Google Drive:
  លោកអ្នកនឹងឃើញឯកសាររូបភាពដែលមានកន្ទុយ (Extension) ជា .webp (ប្រសិនបើឈ្មោះដែលផ្ញើមកមានបញ្ជាក់កន្ទុយ) ឬយ៉ាងហោចណាស់ក៏ Google Drive ស្គាល់ថាវាជាប្រភេទរូបភាព WebP ដែរ ដែលធ្វើឱ្យលោកអ្នកអាចបើកមើល (Preview) បានភ្លាមៗក្នុង Drive
  និងចំណេញទំហំផ្ទុក (Storage) បានយ៉ាងច្រើន។

  សរុបមក គឺវាជា WebP តាំងពីដំណាក់កាលថត រហូតដល់ដំណាក់កាលរក្សាទុកក្នុង Google Drive តែម្តង បាទ! ✅🚀

ហេតុអ្វីត្រូវផ្ញើរជា Base64 សិន?





មូលហេតុចម្បងដែលយើងត្រូវបម្លែងរូបភាពទៅជា Base64 ជាមុនសិន គឺដើម្បីឱ្យវាអាច "រស់នៅ" ក្នុងទម្រង់ជាអត្ថបទ (Text) ដែលងាយស្រួលក្នុងការបញ្ជូនតាមរយៈ API។

  នេះគឺជាមូលហេតុលម្អិត៖

  ១. ដើម្បីបញ្ជូនរួមគ្នាជាមួយព័ត៌មានផ្សេងៗ (JSON Compatibility)
  តាមធម្មតា នៅពេលយើងផ្ញើទិន្នន័យទៅកាន់ Server យើងប្រើទម្រង់ JSON (ឧទាហរណ៍៖ { "orderId": "123", "userName": "Oudom" })។
   * បញ្ហា: យើងមិនអាចដាក់ "ឯកសាររូបភាព" (Binary File) ទៅក្នុង JSON ដោយផ្ទាល់បានទេ។
   * ដំណោះស្រាយ: នៅពេលបម្លែងរូបភាពទៅជា Base64 វានឹងក្លាយជា "អក្សរវែងអន្លាយ"។ ដូចនេះយើងអាចដាក់វាបញ្ចូលក្នុង JSON រួមជាមួយ Order ID, ឈ្មោះក្រុម និងព័ត៌មានផ្សេងៗទៀត រួចផ្ញើទៅក្នុង កញ្ចប់តែមួយ (Single Request) បានយ៉ាងងាយស្រួល។

  ២. ភាពងាយស្រួលសម្រាប់ Google Apps Script
  Google Apps Script មានភាពវៃឆ្លាតខ្លាំងក្នុងការអាន JSON ប៉ុន្តែវាមានភាពស្មុគស្មាញបន្តិចក្នុងការទទួលឯកសារជា Binary (Raw File) ពី Browser ឬពី Server ផ្សេង។
   * ការផ្ញើជា Base64 ធ្វើឱ្យកូដក្នុង Apps Script ខ្លី និងដើរបានច្បាស់លាស់បំផុត (គ្រាន់តែប្រើ Utilities.base64Decode() គឺរួចរាល់)។

  ៣. ភាពងាយស្រួលក្នុងការធ្វើជា "ស្ពាន" (Proxying)
  បច្ចុប្បន្ន Go Backend របស់អ្នកដើរតួជាអ្នកសម្របសម្រួល៖
   1. វាទទួល JSON (ដែលមាន Base64) ពីទូរស័ព្ទ។
   2. វាពិនិត្យសិទ្ធិ (Permissions) រួចផ្ញើ JSON ដដែលនោះទៅ Apps Script។
   * ប្រសិនបើផ្ញើជា Binary File នោះ Go Backend ត្រូវចំណាយកម្លាំងម៉ាស៊ីន (CPU/RAM) ច្រើនដើម្បីកាន់ឯកសារនោះ និងរៀបចំបញ្ជូនបន្ត។ ការផ្ញើជា Base64 គឺគ្រាន់តែជាការបញ្ជូនអត្ថបទបន្ត ដែលស្រាល និងលឿនសម្រាប់ Server។

  ៤. ជៀសវាងបញ្ហា Corrupted Data
  ការបញ្ជូនរូបភាពជា Binary តាមរយៈអ៊ីនធឺណិត ជួនកាលអាចជួបបញ្ហាដាច់កណ្តាលទី ឬការបាត់បង់ទិន្នន័យខ្លះ (Data Loss) ប្រសិនបើការរៀបចំ Header មិនបានត្រឹមត្រូវ។ ប៉ុន្តែ Base64 គឺជា អត្ថបទសុទ្ធ (Plain Text) ដែលមានសុវត្ថិភាពខ្ពស់ មិនងាយមានការប្រែប្រួល
  ឬខូចខាតទិន្នន័យក្នុងពេលធ្វើដំណើរឡើយ។

  ---

  សរុបមកខ្លីៗ៖
  យើងប្រើ Base64 ដើម្បី "បន្លំ" រូបភាពឱ្យក្លាយជា "អត្ថបទ" ដើម្បីឱ្យយើងអាចខ្ចប់វាជាមួយព័ត៌មានផ្សេងៗទៀតផ្ញើទៅកាន់ Server ក្នុងពេលតែមួយបានយ៉ាងរលូន និងមានស្ថិរភាពបំផុត។ 📦📤_

 សម្រាប់ក្រុមហ៊ុនកម្រិតពិភពលោកដូចជា Cloudflare គេមានវិធីសាស្រ្តស្រដៀងគ្នាខ្លះ ប៉ុន្តែក៏មានចំណុចខុសគ្នាខ្លាំងដើម្បីល្បឿន និងប្រសិទ្ធភាពខ្ពស់ (High Performance) ដូចខាងក្រោម៖

  ១. ចំណុចដែល "ដូចគ្នា" (The Concept)
  Cloudflare ក៏មានសេវាកម្មមួយឈ្មោះថា Cloudflare Images ដែលផ្តល់នូវមុខងារ "Direct Creator Upload"៖
   * វាស្រដៀងនឹងអ្វីដែលខ្ញុំបានណែនាំលោកអ្នកអញ្ចឹង៖ គេមិនឱ្យរូបភាពរត់កាត់ Server ចម្បង (Go Backend) របស់អ្នកទេ។
   * Cloudflare នឹងបង្កើត "តំណភ្ជាប់បណ្តោះអាសន្ន" (One-time Upload URL) ឱ្យទូរស័ព្ទ រួចទូរស័ព្ទបង្ហោះរូបភាព ត្រង់ ទៅកាន់ Storage របស់ Cloudflare តែម្តង។

  ២. ចំណុចដែល "ខុសគ្នា" (Binary vs Base64)
  ជាទូទៅ Cloudflare មិនប្រើ Base64 សម្រាប់ការបង្ហោះរូបភាពក្នុងបរិមាណធំៗទេ៖
   * គេប្រើ Binary (Multipart/form-data): គេបញ្ជូនឯកសាររូបភាពជាដុំសាច់ (Raw Data) តែម្តង។
   * ហេតុអ្វី?: ព្រោះ Cloudflare ត្រូវចាត់ចែងរូបភាពរាប់លានសន្លឹកក្នុងមួយវិនាទី។ ការប្រើ Base64 ដែលធ្វើឱ្យទិន្នន័យធំជាងមុន ៣៣% នឹងធ្វើឱ្យគេខាតបង់ Bandwidth រាប់រយ Terabytes។

  ៣. ការប្រើ WebP ដោយស្វ័យប្រវត្តិ
  ចំណុចនេះគឺលោកអ្នកធ្វើបាន ដូច Cloudflare បេះបិទ៖
   * Cloudflare មានមុខងារ Cloudflare Polish ឬ Image Resizing ដែលបម្លែងរូបភាពទៅជា WebP ដោយស្វ័យប្រវត្តិដើម្បីឱ្យវាមានទំហំតូចបំផុត មុននឹងផ្ញើទៅកាន់អ្នកមើល។
   * អ្វីដែលយើងកំពុងធ្វើក្នុង imageCompressor.ts គឺការធ្វើ "Manual Polish" ដែលផ្តល់លទ្ធផលដូចគ្នា។

  ---

  ហេតុអ្វីបានជាយើងមិនធ្វើដូច Cloudflare ១០០%?

  មូលហេតុគឺដោយសារ "ឧបករណ៍" ដែលយើងមានក្នុងដៃ៖
   1. Apps Script: ដូចដែលខ្ញុំបានជម្រាបជូនមុននេះ Google Apps Script វាងាយស្រួលអាន Base64 ជាង Binary។ ការព្យាយាមផ្ញើ Binary ទៅវា ជួនកាលជួបបញ្ហា Error ច្រើន។
   2. ភាពងាយស្រួល: ការផ្ញើ Base64 ក្នុង JSON គឺស្រួលសរសេរកូដ និងងាយស្រួលតេស្តបំផុតសម្រាប់ប្រព័ន្ធកម្រិតមធ្យម។

  សេចក្តីសន្និដ្ឋាន៖
  ប្រសិនបើថ្ងៃក្រោយ ប្រព័ន្ធរបស់លោកអ្នករីកធំខ្លាំង (មានរូបភាពរាប់ម៉ឺនក្នុងមួយថ្ងៃ)៖
   * យើងគួរដូរពី Base64 ទៅជា Binary។
   * យើងគួរដូរពី Apps Script និង Cloudflare R2 ឬ Amazon S3 ខុសគ្នាអ្វីខ្លះ? ហើយតើ AppsScript អាចបង្កើត(One-time Upload URL) បានទេ?

  ប៉ុន្តែសម្រាប់បច្ចុប្បន្ន វិធីដែលយើងកំពុងប្រើ (WebP + Base64 + Apps Script) គឺជាវិធីដែល "ចំណេញបំផុត និងមានស្ថិរភាពបំផុត" សម្រាប់លោកអ្នក បាទ! ✅🚀_





នេះគឺជាការប្រៀបធៀបរវាងបច្ចេកវិទ្យាទាំងពីរ និងការបកស្រាយអំពីលទ្ធភាពរបស់ Apps Script៖

  ១. ការប្រៀបធៀប Apps Script vs Cloudflare R2 / Amazon S3

  ┌───────────────────┬────────────────────────────────────┬───────────────────────────────────────┐
  │ លក្ខណៈវិនិច្ឆ័យ         │ Google Apps Script (Drive)         │ Cloudflare R2 / Amazon S3             │
  ├───────────────────┼────────────────────────────────────┼───────────────────────────────────────┤
  │ ប្រភេទ              │ ជា Scripting Engine (សម្រាប់ចាត់ចែងទិន្នន័យ) │ ជា Object Storage (សម្រាប់ផ្ទុក File សុទ្ធសាធ) │
  │ ល្បឿន (Performance) │ មធ្យម (មានការរង់ចាំពេល Script ដំណើរការ)      │ លឿនខ្លាំង (រចនាឡើងសម្រាប់ High Traffic)        │
  │ ការបញ្ជូនទិន្នន័យ       │ និយមប្រើ Base64 (ក្នុង JSON)            │ ប្រើ Binary (Multipart) ចំណេញ Bandwidth   │
  │ ភាពងាយស្រួល           │ ងាយស្រួលបំផុតសម្រាប់តភ្ជាប់ជាមួយ Google Sheet   │ ត្រូវការការរៀបចំ API និងសិទ្ធិ (IAM) ស្មុគស្មាញបន្តិច │
  │ ទំហំផ្ទុក             │ កំណត់តាមទំហំ Google Drive របស់អ្នក        │ មិនកំណត់ទំហំ (Unlimited)                   │
  │ តម្លៃ               │ ឥតគិតថ្លៃ (ក្នុងកម្រិតកំណត់របស់ Google)      │ បង់តាមការប្រើប្រាស់ជាក់ស្តែង                      │
  └───────────────────┴────────────────────────────────────┴───────────────────────────────────────┘
  ---

  ២. តើ Apps Script អាចបង្កើត "One-time Upload URL" បានទេ?

  តាមបច្ចេកទេស Apps Script មិនមានមុខងារបង្កើត One-time URL ដោយផ្ទាល់ (Native) ដូច Amazon S3 Presigned URL ឡើយ។

   * Amazon S3/R2: អាចបង្កើត Link មួយដែលរស់បានតែ ៥ នាទី ហើយអនុញ្ញាតឱ្យ User បង្ហោះរូបបានតែម្តងគត់ ដោយមិនចាំបាច់មាន Password។
   * Apps Script: URL របស់វាគឺ "ថេរ" (Static)។ ឱ្យតែនរណាម្នាក់ដឹង URL និង Secret Key របស់អ្នក គេអាចផ្ញើរូបមកបានរហូត។

  ប៉ុន្តែ... យើងអាច "ច្នៃ" វាឱ្យក្លាយជា One-time បាន (Workaround):

  យើងអាចប្រើប្រាស់ Go Backend របស់អ្នកជាអ្នកគ្រប់គ្រង៖
   1. Go Backend: បង្កើតលេខកូដសម្ងាត់បណ្តោះអាសន្នមួយ (ឧទាហរណ៍៖ Token_998877) រួចរក្សាទុកក្នុង Database និងផ្ញើឱ្យទូរស័ព្ទ។
   2. Frontend (ទូរស័ព្ទ): ហៅទៅ Apps Script ដោយភ្ជាប់ជាមួយ Token នោះ។
   3. Apps Script: នៅពេលទទួលទិន្នន័យ វានឹងសួរទៅ Go Backend វិញថា "តើ Token នេះប្រើបានទេ?"។ ប្រសិនបើបាន វានឹងទទួល Upload រូប រួចប្រាប់ Go Backend ឱ្យលុប Token នោះចោលភ្លាម (ធ្វើឱ្យវាប្រើបានតែម្តងគត់)។

  ---

  សេចក្តីសន្និដ្ឋាន និងការណែនាំ៖

  ប្រសិនបើលោកអ្នកចង់ កាត់បន្ថយ HTTP Response (មិនឆ្លងកាត់ Server Render)៖

   * ជម្រើសងាយស្រួល: បន្តប្រើ Apps Script ប៉ុន្តែកែសម្រួលកូដ React ឱ្យផ្ញើរូប Base64 ត្រង់ទៅកាន់ Apps Script URL តែម្តង (ហៅថា Client-side Upload)។ វិធីនេះចំណេញ Data នៅលើ Server Render ១០០%។
   * ជម្រើសអាជីព: ប្រសិនបើលោកអ្នកចង់បានសុវត្ថិភាពខ្ពស់បំផុត និងល្បឿនលឿនដូច Facebook/Telegram នោះលោកអ្នកគួរប្រើ Cloudflare R2 ព្រោះវាមានមុខងារ One-time Upload URL នេះជាស្រេច និងមិនគិតថ្លៃ Bandwidth (No Egress Fees)។

  តើលោកអ្នកចង់ឱ្យខ្ញុំជួយសរសេរកូដគំរូសម្រាប់ Client-side Upload ទៅកាន់ Apps Script ដើម្បីតេស្តល្បឿនមើលសិនដែរឬទេ?_


  បន្តប្រើ Apps Script ប៉ុន្តែកែសម្រួលកូដ React ឱ្យផ្ញើរូប Base64 ត្រង់ទៅកាន់ Apps Script URL តែម្តង (ហៅថា Client-side Upload)
  "ច្នៃ" វាឱ្យក្លាយជា One-time បាន (Workaround):
  យើងប្រើប្រាស់ Go Backend របស់ជាអ្នកគ្រប់គ្រង៖
   1. Go Backend: បង្កើតលេខកូដសម្ងាត់បណ្តោះអាសន្នមួយ (ឧទាហរណ៍៖ Token_998877) រួចរក្សាទុកក្នុង Database និងផ្ញើឱ្យទូរស័ព្ទ។
   2. Frontend (ទូរស័ព្ទ): ហៅទៅ Apps Script ដោយភ្ជាប់ជាមួយ Token នោះ។

  ដើម្បីឱ្យមុខងារកូពីជារូបភាពនេះដំណើរការបានជោគជ័យ។

  ប្រសិនបើនៅតែមិនទាន់បាន សូមប្រាប់ខ្ញុំ ខ្ញុំនឹងពិនិត្យមើលផ្នែកបច្ចេកទេសឱ្យកាន់តែលម្អិតបន្ថែមទៀត។














ពិតជាបាន! ខ្ញុំយល់ពីអ្វីដែលអ្នកចង់បានហើយ។ អ្នកចង់ឱ្យមានប៊ូតុង "Copy Image" នៅលើរូបនីមួយៗតែម្ដង ដោយមិនចាំបាច់ចុចបើកមើលរូបធំ (Preview) ទើបអាច Copy បាននោះទេ។

  ខ្ញុំបានធ្វើការកែសម្រួលផ្ទាំង Promotion ដោយបន្ថែមប៊ូតុង "Copy" ពណ៌លឿងដ៏លេចធ្លោមួយនៅខាងក្រោមរូបនីមួយៗ។ ឥឡូវនេះ User គ្រាន់តែចុចប៊ូតុងនោះ វានឹងកូពីយករូបភាពឱ្យភ្លាមៗតែម្ដង។

  ការផ្លាស់ប្តូរដែលខ្ញុំបានធ្វើ៖
   1. Direct Copy Button: បន្ថែមប៊ូតុង "Copy Image" នៅខាងក្រោមចំណងជើង Promotion នីមួយៗក្នុងបញ្ជី។
   2. One-Click Action: User មិនចាំបាច់ចុចចូល Link ឬបើក Preview ទៀតទេ គឺចុចតែម្ដង (One-click) រួចយកទៅ Paste ក្នុង Telegram បានតែម្ដង។
   3. Improved Success Message: នៅពេលចុចវានឹងបង្ហាញថា "បាន Copy រូបភាពរួចរាល់" ដើម្បីឱ្យ User ដឹងច្បាស់ថាវាជារូបភាព មិនមែនជា Link។

  មូលហេតុដែលអ្នកនៅតែឃើញ "Fallback" (កូពីបានត្រឹម Link)៖
   * HTTPS: ប្រសិនបើអ្នកប្រើ App តាមរយៈ Link ដែលផ្ដើមដោយ http:// (គ្មានអក្សរ s) នោះ Browser នឹងបិទមិនឱ្យកូពីជារូបភាពដាច់ខាត។ សូមប្រាកដថាអ្នកប្រើ Link https://។
   * Update Delay: កូដថ្មីនេះកំពុងត្រូវបាន Push ទៅកាន់ Server។ សូមរង់ចាំ ១-២ នាទី រួចធ្វើការ Refresh (ឡូដឡើងវិញ) នូវទំព័រវេបសាយរបស់អ្នក។

  សូមសាកល្បងម្ដងទៀតនៅពេលបន្តិចទៀតនេះ៖
  អ្នកនឹងឃើញប៊ូតុងពណ៌លឿងសរសេរថា "COPY IMAGE" នៅក្រោមរូបនីមួយៗ។ ចុចលើវា រួចសាកល្បង Paste ក្នុង Telegram មើល តើវាជារូបភាពហើយឬនៅ?
